这部分介绍的是：现在的是我是按照我学长提出的一套ai写代码的开发工作流程，如你所见，我们这一整套都是Claude写的。
他的核心思想是这样的一套流程：Initial Requirement List -> AI for Polishing and Expanding (ChatGPT/Gemini etc) <-> Human Review -> Transform to Json file -> Specially Designed Code Agent -> Human Review
我学长的思路是，我现在做的这个的一个重要意义：论证他的这一套流程的可行性和效率。我觉得我这套代码的复杂程度应该是他这套的实践最高成果了。所以请考虑这个逻辑，我打算放在介绍项目代码和结果之前讲，感觉就“我们这套流程是这样，让我们来看看我们现在做出了的这个项目，这个项目做的还可以，说明这套工作流也很可以”
这套工作流程他在之前的组会上展示过了，我们就不要在讲了，我提一嘴他讲过就行
现在我之所以要将这个部分，一方面是声明一下我用了ai，一方面是我也有点新的感悟基于他的这套工作流：
他这套的一个关键薄弱点在于，最后生成的可视化结果必须由人来review并反馈给ai问题让他改进。但这个过程中，1、人的描述可能十分偏离，有侧重，ai无法正确理解，这里面的信息损耗极为严重；2、在仿真开发中，现象的描述可能会及其困难，尤其是当遇到一些很奇怪的问题时，人根本不知道如何描述；3、截图给ai，可以，但它基于图片识别的能力还是很薄弱，大多数时候都无法理解，对数值问题他更是无法获取。
所以我们最直观的想法就是让ai自己查看仿真生成的结果，但这又有问题：1、仿真数据动辄一帧就几个G，特别是3D结果，AI根本处理不了，即使能，也会消耗惊人的token；2、类似Gemini根本无法读取vtk这些文件数据；3、即使对于小数据量的文件，文本ai对于这种仿真数据无法像人一样可视化分析，无法获取到关键信息和问题所在

所以我的一个小改进是：让一个专职的agent分析数据，它的工作不是读取vtk，而是写python和脚本读取关心的关键信息，提取温度、速度数据范围、分布、占比，或者界面形状、剖面形状，固液气相的占比和方位等等。
当然，人的参与总是必须的——软件和仿真最终还给人看的，所以最后那一步Human Review还是不可缺少的，不过人的工作大大简化：跑完一个仿真后，自动先让vtk分析agent写脚本分析关键帧的基础数据，比如速度温度或者最近出问题比较多的一些参数或者几何；然后人再打开vtk检查，根据自身需求和观察提出问题和错误（结果分析的脚本也会经常出错，并且分析的信息量终归是不如经验丰富的cfd工程师的眼睛），告诉vtk分析agent哪一方面有问题，是要着重提取哪个中心界面的形状、激光温度分布的异常、等等等等，它会自己更改脚本以作出最适合写代码的agent的理解的分析和判断。

这里面有个关键点：一定不要让ai自己做数据处理，ai的计算准确性有目共睹，经常会出现一些及其离谱的偏差，搞错单位，搞错数量级，搞错变量名。所以所有涉及计算的数据处理必须交由他写的脚本来完成，这也是为什么要设置这样的一个分析agent--它不是来分析数据的，而是规划分析数据的方法，并且写代码来分析数据

在面对小规模的项目时候，我这个改进必要性没太大，毕竟要么人能很容易的跟ai描述结果和问题，或者ai能自己分析结果。但在命令有大量变量、参数和数据量、复杂三维结构的时候，这将是唯一的办法
